# 대시보드 시트 숫자 포맷 수정 PRD

## 배경

대시보드 시트에 데이터는 정상적으로 기록되지만, 숫자 포맷이 올바르게 적용되지 않는 문제가 발생하고 있다.
기존에 수동으로 포맷을 적용해놓은 상태에서, 코드가 시트를 초기화(`clear_sheet`) 후 재작성할 때
`numberFormat`이 일부 셀에 정확히 적용되지 않거나 잘못된 포맷이 적용되는 현상이다.

## 현재 발생하는 문제

### 문제 1: 수익률(%) 컬럼에 ₩ 포맷 적용 (월별 성과 섹션)

| 위치 | 현재 표시 | 기대 표시 |
|------|----------|----------|
| 2026-03 IRP 수익률 | ₩0 | 0.00% |
| 2026-03 ISA 수익률 | ₩0 | 0.00% |
| 2026-03 주식1~3 수익률 | ₩0 | 0.00% |
| 2026-03 해외주식1 수익률 | ₩0 | 0.00% |

**원인 분석**: `_apply_dashboard_formats()`에서 월별 성과 섹션의 H열(수익률)에 `0.00%` + `PERCENT` 타입을 적용하지만, 2026-01/02 행에는 정상 적용되고 2026-03 행에는 적용되지 않음. 포맷 적용 범위(`monthly_start+1` ~ `stock_start-2`) 계산에 문제가 있거나, `clear_sheet`가 기존 수동 포맷을 완전히 제거하지 못하는 것으로 추정.

### 문제 2: 통화 포맷 누락 (월별 성과 섹션)

| 위치 | 현재 표시 | 기대 표시 |
|------|----------|----------|
| 2026-03 IRP 실현손익 | 25,565 | ₩25,565 |
| 2026-03 ISA 실현손익 | 33,301 | ₩33,301 |
| 2026-03 해외주식1 매수금액 | 17252397 | ₩17,252,397 |

**원인 분석**: 동일하게 포맷 적용 범위 문제. 2026-03 해외주식1은 천단위 구분자(,)도 없어 포맷이 전혀 적용되지 않은 상태.

### 문제 3: 섹터별 투자비중에 잘못된 포맷 (투자 지표 섹션)

| 위치 | 현재 표시 | 기대 표시 |
|------|----------|----------|
| 산업재 | 0.02263933281 | 2.26% |
| 에너지~필수소비재 (값=0) | ₩0 | 0.00% |

**원인 분석**: `_apply_metrics_formats()`에서 `pct_offsets`로 퍼센트 포맷을 적용하는데, 산업재는 포맷이 아예 적용되지 않고(raw decimal), 값이 0인 행들은 ₩ 포맷이 적용됨. 행 오프셋 계산 또는 포맷 적용 순서 문제.

### 문제 4: 투자 지표 내 기타 퍼센트 셀에 ₩ 포맷

| 위치 | 현재 표시 | 기대 표시 |
|------|----------|----------|
| 상위 5종목 집중도 | ₩0 | 0.00% (또는 실제 비율) |
| 평균 수익률 | ₩0 | 0.00% |
| 평균 손실률 | ₩0 | 0.00% |

### 문제 5: 수익/손실 Top 10에 잘못된 포맷

| 위치 | 현재 표시 | 기대 표시 |
|------|----------|----------|
| 삼성전자우 (수익 Top) | 384042 | ₩384,042 |
| AMD (손실 Top) | -23468000.00% | -₩234,680 |
| 엔비디아 (손실 Top) | -21773100.00% | -₩217,731 |
| 기타 손실 Top 종목 | -351914 등 | -₩351,914 등 |

**원인 분석**: AMD, 엔비디아는 퍼센트 포맷이 잘못 적용되어 KRW 값이 `%`로 표시됨(-234680 → -23468000.00%). 나머지 손실 Top 종목과 삼성전자우는 포맷이 전혀 적용되지 않음. `krw_offsets` 계산이 일부 행만 커버하거나 pct/krw 포맷이 뒤섞인 것으로 추정.

## 근본 원인 추정

### 1. `clear_sheet`는 값만 지우고 포맷은 남김

`clear_sheet(start_row=1)`은 Google Sheets API의 `values().clear()`를 사용하여 **값만 삭제**한다.
기존에 수동으로 적용된 `numberFormat`(₩ 통화 등)은 셀에 그대로 남아있어,
새로 값을 쓸 때 이전 포맷이 계속 적용된다.

### 2. `repeatCell`로 포맷을 덮어씌우지만 범위가 부족할 수 있음

`_apply_dashboard_formats()`의 범위 계산:
```python
monthly_data_end = stock_start - 2  # 실제 데이터 마지막 행
```
이 계산이 정확한지 검증 필요. 빈 행(separator) 포함 여부에 따라 마지막 몇 행이 누락될 수 있음.

### 3. `_apply_metrics_formats`의 offset 계산 문제

투자 지표/매매 인사이트 섹션은 행별로 pct/krw 포맷을 개별 적용하는데,
`pct_rows.append(len(rows))` 시점과 실제 데이터 행 위치 사이에 off-by-one 가능성:
```python
pct_rows.append(len(rows))          # offset = 현재 rows 길이
rows.append([label, value])         # 이 행이 실제 rows[len(rows)] 위치
# → abs_row = start_row + len(rows) (append 전의 len)
# → sheet에서 start_row + offset 행에 데이터가 위치
```
이 로직 자체는 맞아 보이지만, 동적으로 섹터 섹션이 추가/생략될 때 offset이 틀어질 수 있음.

### 4. 포맷 적용 순서 충돌

`_apply_metrics_formats`에서 pct 포맷을 먼저 적용하고 krw 포맷을 나중에 적용한다.
만약 같은 행이 양쪽 offset에 잘못 등록되면, krw 포맷이 pct 포맷을 덮어씌운다.
